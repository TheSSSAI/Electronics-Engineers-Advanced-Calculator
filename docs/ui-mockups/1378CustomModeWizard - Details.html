Description

A multi-step workflow component that guides a user through the process of creating or editing a custom calculation mode. It orchestrates several molecular and atomic components to manage state across multiple steps.

Design System Alignment

This organism composes molecular components like WizardStep and VariableListManager, and atomic components like Button, TextInput, and FormulaEditor. It uses the design system's spacing for layout, colors for states, and typography for hierarchy.

Responsive Specifications

Mobile Adaptations

  • The wizard becomes a full-screen component, occupying the entire viewport.
  • Padding is reduced to maximize content space.
  • The stepper component within the wizard hides its text labels, showing only numbered indicators.

Tablet Adaptations

  • Maintains the desktop layout but within a constrained width, allowing the page background to be visible.

Desktop Enhancements

  • Displayed as a large, centered card or modal with a clear shadow, giving it prominence.
  • Full step labels are visible in the stepper.

Accessibility Features

Keyboard Support

The entire wizard is navigable via keyboard. Focus moves logically from the stepper to the content and finally to the footer buttons.

Screen Reader Support

The stepper is a `

`. The active step is marked with `aria-current='step'`. Content panels are associated with their step controls. Loading and error states are announced via ARIA live regions.

Focus Management

When navigating between steps, focus is programmatically moved to the first focusable element (e.g., the `h3` title or first input) of the new step's content.

Aria Attributes

  • aria-label for the stepper nav
  • aria-current='step'
  • role='tabpanel' for content areas

Color Contrast

All text and interactive elements meet WCAG 2.1 AA contrast requirements.

Implementation Notes

Embedded Css Architecture

All CSS for the wizard organism and its dependent molecular/atomic components are embedded in the HTML head. A mobile-first approach is used with BEM naming conventions.

Component Organization

The HTML file showcases the wizard in its primary variants (create, edit) and demonstrates each step and state (saving, error) in clearly labeled sections.

Responsive Design

Flexbox is used for the main wizard layout. Media queries adjust padding and stepper visibility for smaller screens.

Accessibility Compliance

Semantic HTML, ARIA attributes for state management, and robust focus management are implemented to ensure WCAG 2.1 AA compliance.

Browser Support

Targets modern browsers, utilizing widely supported CSS features like flexbox and custom properties.

Usage Guidelines

When To Use

Use for guiding users through a required, linear sequence of tasks, such as creating a new resource.

Content Guidelines

Each step should focus on a single, coherent task. Step titles should be concise. Footer buttons should have clear, action-oriented labels.

Placement Rules

Should be used as the primary content of a page or within a large modal that overlays the application.

Customization Options

  • The number and content of steps are fully configurable.
  • Can be initialized in 'create' or 'edit' mode, which alters titles and pre-populates data.

Testing Recommendations

Browser Testing

Test in Chrome, Firefox, Safari, Edge.

Device Testing

Verify layout and interactivity on mobile (375px), tablet (768px), and desktop (1280px) viewports.

Accessibility Testing

  • Full keyboard-only navigation through all steps.
  • Screen reader testing (NVDA/VoiceOver) to confirm step changes and states are announced.
  • Focus management verification when navigating between steps.

Performance Testing

Ensure smooth transitions between steps with no noticeable lag.
