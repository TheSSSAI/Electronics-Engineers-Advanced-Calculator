# 1 Overview

## 1.1 Diagram Id

SEQ-DF-001

## 1.2 Name

Offline Data Modification and Synchronization

## 1.3 Description

A logged-in user modifies their data (e.g., adds a calculation to history, saves a variable) while offline. The client queues these changes locally in IndexedDB. When network connectivity is restored, the client automatically syncs the queued changes with the backend, using a last-write-wins strategy for conflicting updates.

## 1.4 Type

üîπ DataFlow

## 1.5 Purpose

To provide a seamless user experience by allowing data modifications during periods of no connectivity and ensuring data consistency once reconnected, as per REQ-ENV-001.

## 1.6 Complexity

High

## 1.7 Priority

üî¥ High

## 1.8 Frequency

OnDemand

## 1.9 Participants

- REPO-SPA-CLIENT
- REPO-API-GATEWAY
- REPO-APP-USER-DATA
- REPO-DB-POSTGRES

## 1.10 Key Interactions

- Client SPA detects it is offline.
- User performs an action that modifies data (e.g., saves a variable with a client-side timestamp).
- The client-side service intercepts the API call and stores the modification request in an IndexedDB queue.
- Client SPA detects network connectivity is restored.
- The client-side sync manager processes the queue, sending each request to the User & Data Service.
- For append-only data (history), items are added.
- For mutable data (variables), the backend compares the incoming timestamp with the existing record's 'updatedAt' timestamp and applies a 'last-write-wins' strategy, with the server's state being authoritative in a conflict.
- Upon successful sync, the item is removed from the local IndexedDB queue.

## 1.11 Triggers

- User modifies data while offline, and then later reconnects to the internet.

## 1.12 Outcomes

- User data created or modified offline is successfully persisted to the backend.
- Data consistency is maintained between client and server.

## 1.13 Business Rules

- Append-only data (history) is merged.
- Mutable data (variables, custom mode definitions) uses a 'last-write-wins' conflict resolution strategy based on server-authoritative timestamps.

## 1.14 Error Scenarios

- Sync fails for a specific item (e.g., validation error); the item is marked as failed in the queue for manual user intervention.
- Backend is unavailable during sync attempt; client will retry later.

## 1.15 Integration Points

- Browser IndexedDB API.

## 2.6.0.0.0.0.0 Notes

### 2.6.1.0.0.0.0 Content

#### 2.6.1.1.0.0.0 Content

Offline detection relies on the browser's `navigator.onLine` API, which indicates connection to a network, not necessarily the internet. The sync process must handle server unavailability gracefully.

#### 2.6.1.2.0.0.0 Position

Top

#### 2.6.1.3.0.0.0 Participant Id

REPO-SPA-CLIENT

#### 2.6.1.4.0.0.0 Sequence Number

2

### 2.6.2.0.0.0.0 Content

#### 2.6.2.1.0.0.0 Content

The 'last-write-wins' (LWW) logic is implemented by comparing the client's modification timestamp with the server's 'updatedAt' timestamp. A pessimistic lock (`SELECT FOR UPDATE`) is used within a transaction to prevent race conditions during the read-compare-write cycle.

#### 2.6.2.2.0.0.0 Position

Right

#### 2.6.2.3.0.0.0 Participant Id

REPO-APP-USER-DATA

#### 2.6.2.4.0.0.0 Sequence Number

### 2.6.3.0.0.0.0 Content

#### 2.6.3.1.0.0.0 Content

Append-only data like 'calculation_history' does not require LWW logic and can be handled with a simple INSERT, making its synchronization flow simpler than the one shown for mutable data.
